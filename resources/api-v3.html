Payments Connector

Version 3.0.0  ·  Meridian Bank NV, Amsterdam  ·  SCT + SCT Inst + SWIFT

Adds SWIFT (cross-border) to the v2 baseline. SWIFT support required significant expansion of the request contract: correspondent agent, charge bearer, purpose code, remittance info, SWIFT gpi reference, and instruction priority are all new. The request now contains 20+ fields, closely mirroring ISO 20022 pain.001.

What changed from v2: Request adds 8 new fields for SWIFT. paymentMethod now supports SWIFT. The failure response is unchanged from v1 and v2. This is the gap.
The asymmetry: The request side grew from 7 fields (v1) to 12 (v2) to 20+ (v3), carefully tracking ISO 20022 pain.001 with each new rail. The failure response has been status + optional errorCode since v1. No origin field has ever been added. This is the specification gap identified in UAT cycle 3 at Meridian. It is addressed in v3.5.
Base URL https://api.grandcentral.backbase.io/v3
POST /payments/initiate Initiate a payment

Routes to SCT, SCT Inst, or SWIFT based on paymentMethod. The core names the rail; the connector translates, submits, and returns. When SWIFT is named, the connector formats an MT103 message for the SWIFT network via the debtorAgent's SWIFT gateway.

The specification gap - as identified in Meridian UAT cycle 3

The failure response envelope has not changed since v1. Three failure categories are not actionable by ops:

Core errors: T24 codes (e.g. AC.CLOSED) pass through raw. The connector has no translation mapping. Patched at Apex Banking in v1 with a bank-specific transformer - not in this baseline.

Connector timeouts: When SCT Inst returns no response in 10 seconds, the connector returns an empty body. Patched at Sparkasse in v2 - not in this baseline.

SWIFT rejections: SWIFT rejection responses are not standardised in this contract. They return as PAYMENT_FAILED with an empty message. This failure mode is new in v3 and has not been patched anywhere.

Fix: v3.5 adds an origin field (derived from the config mapping, not runtime state) and makes errorCode a canonical translated value, not a passthrough. See gc_error_mapping_config_v3.json for the full mapping. The fix requires no code change to the connector - only a config expansion and a contract version bump.
